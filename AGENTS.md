# Agent Notes

## Project Intent

This repo owns a private cinematic war room wallboard for a single landscape TV. The first screen should be the working wallboard experience, not a marketing page. Keep the interface operational, dense, dark, readable from across the room, and free of visible organization branding.

## Implementation Conventions

- Prefer configuration-driven panels and server-side data fetching.
- `public/index.html` is the static single-file wallboard (moved from repo root 2026-07-09 so Next.js serves it verbatim at `/index.html` on whatever host runs this app). It defaults to fetching `/api/wallboard` same-origin, and may still be pointed at another JSON URL with `?api=...` (e.g. a plain `wallboard.json` for a fully static, backend-less mirror hosted elsewhere). It must not contain provider credentials, private keys, GA service account material, Apify tokens, or secret dashboard/API keys. `wallboard.json` is gitignored; `wallboard.example.json` is the checked-in data-shape reference.
- Static hosting does not remove the need for a secure server-side wallboard API: GA, Apify, monitor status, website/SSL checks, YouTube live detection, and any other credentialed/provider-sensitive calls stay behind that API or another secret-managed backend. Never move those secrets into browser JavaScript to make static hosting "work." **This was tested directly during the 2026-07-09 public-hosting design pass** — a request to embed raw GA/Apify credentials into `index.html` was deliberately declined (it would leak long-lived credentials to every visitor/crawler) even after the requester confirmed they understood the risk and wanted it anyway. Treat this as settled; do not re-litigate it on a future ask without the user explicitly overriding this file's own rule in writing first.
- Production is now a **single Cloudflare Worker deployment** (decided 2026-07-16 after rejecting Railway's paid hosting): `wrangler.jsonc` serves `public/index.html` from the static-assets binding and `cloudflare/worker.js` implements the secret-bearing `/api/wallboard` route itself. Provider code lives in `cloudflare/analytics.js`, `cloudflare/providers.js`, `cloudflare/config.js`, and `cloudflare/cache.js`. Never move those secrets into browser JavaScript; the Cloudflare runtime is the server-side security boundary. The Next.js `/wallboard` and `app/api/*` routes remain the local/full-app surface and a behavior reference, not a required production host.
- The production URL is `https://data-war-room.lboone.workers.dev/`. Git auto-deployment from `lboone-bc/data-war-room` was restored and verified on 2026-07-16; the GitHub App installation must retain access to both `data-war-room` and the companion `cctv-weather` repository. There is no `WALLBOARD_API_ORIGIN` in the Cloudflare-only architecture.
- The Cloudflare repository link was explicitly disconnected/reconnected on 2026-07-16 after it stopped reacting to pushes. Its current production branch is `main`, production deploy command is `npm run deploy:cloudflare`, and non-production builds are disabled. If a future push does not create a build, inspect the Worker Settings build section for the disconnected warning before changing code or secrets.
- Production GA was configured and live-verified 2026-07-16 using `GA_PROPERTY_ID` plus the encrypted `GOOGLE_APPLICATION_CREDENTIALS_JSON` Worker secret; the service account has working access to the configured GA4 property. Never commit the downloaded service-account JSON or copy its private key into documentation. Cloudflare Git deploys must use `npm run deploy:cloudflare` (the repo-pinned Wrangler), not bare `npx wrangler deploy`: Cloudflare's latest auto-config path incorrectly tried to convert this static-assets Worker into an OpenNext app and rejected the repo's Next.js 14 diagnostic/local surface. Keep `"keep_vars": true` in `wrangler.jsonc`; without it, a Git/CLI deployment overrides and removes dashboard-managed runtime variables, including the GA secret (reproduced directly 2026-07-16).
- Configure production under Cloudflare **Workers & Pages → data-war-room → Settings → Variables and Secrets**. Sensitive values (`WALLBOARD_ACCESS_TOKEN`, `GA_PRIVATE_KEY` or `GOOGLE_APPLICATION_CREDENTIALS_JSON`, `DRIVENC_API_KEY`, `APIFY_TOKEN`, and token-bearing dashboard/monitor URLs) must be encrypted Secrets. `GOOGLE_APPLICATION_CREDENTIALS=/path/file.json` is Node/local-only and cannot work in Cloudflare; use the JSON secret or `GA_CLIENT_EMAIL` + `GA_PRIVATE_KEY` there. `.dev.vars` is the ignored local Worker equivalent. Verify secrets after Git deploys because the dashboard integration previously wiped a secret in the companion project.
- Cloudflare's free plan is the intended production tier: one signage screen is far below its 100,000 requests/day allowance. Keep the 25-second composed-payload edge cache plus provider TTLs/in-flight de-duplication/last-good fallback in `cloudflare/cache.js`; they are required both for free-tier CPU/request efficiency and upstream quota safety.
- Wrangler is intentionally pinned to `~4.63.0`, the newest pre-4.64 line compatible with this repo's Node 20 pin and outside a later esbuild advisory range. Current Wrangler releases require Node 22. Upgrade Wrangler only as part of the future coordinated Node 22 + Next.js 16/React migration; do not silently break the local Node 20 contract just to chase the CLI latest tag.
- Integrations that silently degrade on failure (GA in `lib/analytics.ts`, and by the same pattern Apify/DriveNC/weather/YouTube) intentionally never put raw provider errors in the public API response, but they must log sanitized diagnostic detail server-side via `console.error`. Keep last-good/empty fallbacks and add failed-attempt backoff so provider outages cannot turn the 30-second wallboard poll into an upstream request storm.
- Keep `/wallboard` suitable for unattended Apple TV signage/browser use.
- Preserve graceful degradation: missing credentials, blocked embeds, or API failures should show setup/degraded states rather than crash.
- GA traffic sources should use a standard GA report such as today's `sessionDefaultChannelGroup`; GA realtime rejects source/medium dimensions.
- The world access map uses GA realtime `countryId` plus `country`; keep `COUNTRY_COORDINATES` / `CITY_COORDINATES` aligned in both `lib/analytics.ts` and `cloudflare/analytics.js` when adding precise placements.
- Iframe dashboards are fragile. Check vendor support for kiosk/embed URLs before assuming a normal logged-in page can render inside the wallboard.
- The Site24x7 embed is scaled through `DATABASE_FRAME_VIEWPORT_WIDTH` and `DATABASE_FRAME_VIEWPORT_HEIGHT`; use the current `1024x640` default unless visual QA shows clipping.
- The `DatabaseFrame` iframe's auto-scale calculation (`updateScale` in `app/wallboard/page.tsx`) only has to reserve the `.frame-shell`'s own padding now — the "All Monitors" / "Website Check" / "SSL" stat boxes (`.database-frame-summary`), the refresh/last toolbar, and the ticker were removed from this column (ticker moved to a full-width `.ticker-bar`; toolbar removed 2026-07-09). If a new element is ever added inside `.frame-shell` below the iframe, its rendered height must go back into this reservation math (via a ref + `ResizeObserver`, as the old summary/log entries did) or the iframe will overlap/clip it.
- Site24x7 public dashboard dark theme is configured in Site24x7 share settings, not by styling the iframe from this app.
- Do not embed YallBot/YouTube for the ambient "emergency operations center" feel — that's realized natively instead, via the `SystemLog` component (real alerts + latest Instagram/Facebook posts + heartbeat lines) in `app/wallboard/page.tsx`; treat that as the actual implementation of the style inspiration, not a placeholder for a future embed. `SystemLog` renders as a horizontally scrolling ticker (`.ticker-track`/`.ticker-entry` in `app/globals.css`), not a stacked list — the underlying `useSystemLog` hook logic (entry state, dedup, ambient rotation) is unchanged, only the render/CSS layer is a marquee. Scroll speed is held roughly constant regardless of content length by measuring `scrollWidth` and deriving `animationDuration` from a fixed px/sec constant (`TICKER_PX_PER_SECOND`), not a fixed duration. The ticker itself lives outside `wall-grid` now, in a full-width `.ticker-bar` between the grid and the bottom viewport edge (moved 2026-07-05 from inside the Active Database System panel; footer telemetry moved into `.header-stats` and the footer was removed 2026-07-09). `.wall-grid`'s `height: calc(100vh - Npx)` (both the base rule and the shrunk-viewport breakpoint) reserves `.ticker-bar`'s margin + height on top of the header/incident/wallboard padding reservation — keep those numbers in sync if `.ticker-bar`'s sizing changes. This is distinct from the `LiveStreamPanel` (below), which does embed YouTube directly — that panel shows the org's actual live broadcast for a real operational reason (is the service airing right now), not ambient decoration, so the "no embed" rule doesn't apply to it.
- The ticker's ambient content used to be Hacker News + Fox News headlines (`lib/newsFeed.ts`, both free/no-key public feeds); that was fully replaced 2026-07-05 with the church's own latest Instagram/Facebook posts per an explicit request, and `lib/newsFeed.ts` was deleted (along with the now-unused `fast-xml-parser` dependency) rather than left as dead code. Neither Instagram nor Facebook has a free public feed — confirmed by direct testing that anonymous requests get nothing usable (Facebook returns a flat error page, Instagram returns an empty JS shell) — so this goes through Apify (`lib/apify.ts` locally, `cloudflare/providers.js` in production) using `APIFY_TOKEN` and the `apify~instagram-post-scraper` / `apify~facebook-posts-scraper` actors. Apify is optional and is the only integration here that can create a separate provider bill: leave `APIFY_TOKEN` unset for a truly $0 wallboard. The Cloudflare production cache is one hour per platform; the local Next route retains its 15-minute cache. Failed runs retain last-good content instead of retrying on every poll. Social notices never touch `payload.alerts` or the audible-alert system.
- The "serious" alarm (`database-monitors-down` only — i.e. the Site24x7 All Monitors feed reporting a down monitor) plays a distinct square-wave siren tone (`playAlarmPulse`, `app/wallboard/page.tsx`) on a fixed 12s repeat with no cooldown gate, separate from the normal `playTone`/180s-cooldown path used by every other critical alert (website health, SSL, traffic zero/spike/drop). It stops automatically once `database-monitors-down` drops out of `payload.alerts` — there is no acknowledge/silence control, and it still requires the same `arm audio` gesture as the rest of the audio system. The repeat-siren `useEffect` must key off a stable boolean (`seriousAlertActive`), not the `alerts.find(...)` object itself — every 30s poll (`POLL_MS`) produces a brand-new `alerts` array/object even when nothing changed, so depending on the object reference tears down and restarts the `setInterval` every poll, breaking the fixed 12s cadence into an irregular 12/12/6s pattern (fixed 2026-07-05).
- Map dots must be keyed by more than coordinates: `mapPointKey` (`app/wallboard/page.tsx`) includes `city ?? region` alongside `countryCode`/lon/lat because several distinct GA realtime regions can legitimately share one fallback coordinate — any city not present in `COUNTRY_COORDINATES` collapses to the same country-level anchor (see the world-map bullet above). Keying on coordinates alone let a `Map` silently collapse multiple distinct regions onto a single flickering dot, losing all but the last one in poll order (fixed 2026-07-05, confirmed live with four Biltmore-area cities sharing the US country anchor).
- **Any new external API call (GA, Apify, DriveNC, or future additions) must follow the caching discipline in `cloudflare/cache.js` and the matching `lib/*` implementation: a generous TTL cache, in-flight-promise de-duplication so concurrent requests share one outstanding fetch, and a silent fallback to the last-good/empty result on failure — never throw or surface a raw provider error to the UI.** The Cloudflare Cache API retains provider records for stale-on-error use across warm-isolate restarts; its timestamps, not HTTP cache eviction alone, decide freshness. This was learned the hard way: an early version polled GA Realtime aggressively enough to burn ~14,000 tokens in a clock-hour. GA makes two Realtime reports at most once per five minutes and two standard ranking reports at most once per 15 minutes.
- Local Arden weather (added 2026-07-06) comes from the no-key National Weather Service API using the Arden, NC point `35.4665,-82.5165`; implementations are `lib/weather.ts` locally and `cloudflare/providers.js` in production. Both use a 10-minute TTL and degraded/last-good fallback. The wallboard shows current conditions plus the next three daytime forecast cards in the right-side `.ops-column`.
- The header title is "War Room" (renamed from "Monitoring Room" on 2026-07-09). Header status chips (`.header-stats`) now carry the old footer telemetry: live/degraded mode, optional analytics message, API age, and generated time. The footer element was removed to give the lower panels more height.
- The header also includes a `Rock Update!` countdown (`ROCK_UPDATE_TARGET` in `app/wallboard/page.tsx`) ending at `2026-07-27T00:00:00-04:00`.
- Traffic Detail's Top Pages/Sources are same-day standard GA reports cached 15 minutes, not realtime rows. `analytics.fetchedAt`/`cacheSeconds` describe that rankings cache and must not be replaced with the 30-second wallboard poll timestamp. Realtime summary/map data is cached five minutes.
- Traffic cameras were replaced 2026-07-16 with eight live NCDOT HLS feeds discovered server-side through the DriveNC developer API (`DRIVENC_API_KEY`); the key never reaches the browser. Keep the duplicated curated ID/label/priority configuration aligned in `lib/trafficCameras.ts` and `cloudflare/providers.js`; both cache metadata for 90 seconds and retain last-good/fallback values. `app/wallboard/page.tsx` and `public/index.html` use native HLS on Safari and `hls.js` elsewhere; a feed that does not reach `playing` within 18 seconds falls back to a scaled `https://www.drivenc.gov/map/Cctv/<id>` iframe. The old JPEG proxy and IPCamLive snapshots are gone. Labels remain accessibility text only.
- Production `DRIVENC_API_KEY` is configured as an encrypted Cloudflare Worker secret (verified 2026-07-16); never commit or document its value. The static production header must retain `#fullscreenButton`, since the TV uses that surface instead of the React `/wallboard` header. It supports the standard Fullscreen API, WebKit's legacy API, and the `F` keyboard shortcut; unsupported signage browsers must use their app-level kiosk mode.
- Production `DATABASE_DASHBOARD_URL` (encrypted secret), `YOUTUBE_LIVE_CHANNEL_HANDLE`, and `YOUTUBE_FALLBACK_CHANNEL_HANDLE` were configured and API-verified 2026-07-16. The configured primary channel was offline during verification and the fallback channel was active, confirming both handles were being read. Never document the dashboard URL value; it may contain a share token.
- Panels with a fixed-height budget that can shrink (e.g. via a parent grid's `fr` sizing) should gracefully hide non-essential sub-content rather than render an unreadably squeezed sliver. Prefer CSS container queries scoped to the panel, not viewport media queries, for future panel content that depends on space freed up by other layout changes.
- The same rule applies to text sizing, not just visibility: never size text with `vw` inside deeply nested grid layouts. Website Pulse now only shows two square GA realtime tiles (`Active last 30m` and `GA events last 30m`), and those `.pulse-metrics strong` values are sized off each tile's own rendered width (`cqw`) with `overflow: hidden` as a safety net.
- `LiveStreamPanel` (`app/wallboard/page.tsx`) shows the org's YouTube channel (`YOUTUBE_LIVE_CHANNEL_HANDLE`) live in a muted, autoplaying embed when it's actively broadcasting, with a "LIVE" badge; otherwise a quiet "Not currently live" state. Live/offline detection (`lib/youtubeLive.ts`) needs no YouTube Data API key: it scrapes the channel's `/<handle>/live` page and checks whether the `<link rel="canonical">` tag points at a `watch?v=` URL (only true while actually live), cached 45s with the same in-flight-dedup/silent-fallback discipline as every other external call in this repo (see the bullet below). Use the `@handle` path, not `/channel/<id>/live` — confirmed via direct curl that the channel-ID path is unreliable for at least one real channel (stayed on the channel page while genuinely live) while the handle path resolved correctly for every channel tested. Missing `YOUTUBE_LIVE_CHANNEL_HANDLE` means the panel doesn't render at all (`payload.liveStream.enabled === false`), not a setup warning — same convention as `DATABASE_MONITORS_STATUS_URL`. The embed is muted via the `mute=1` URL param baked into the iframe `src`, not the Web Audio `arm audio` gate used elsewhere — it's ambient/visual-only and was never routed through that system. It originally occupied the one cell the shared 3-row `wall-grid` left open below the database panel, but that gave it only ~23% of the column's height against the database panel's ~77% — see the `.database-column` note below for why that changed (2026-07-05).
- If the primary channel isn't live, `LiveStreamPanel` checks a second channel (`YOUTUBE_FALLBACK_CHANNEL_HANDLE`, defaults to `@livenowfox`) and shows it instead if that one's live, with a distinct amber `.livestream-badge.fallback` badge reading "live now — [source]" so it never reads as "Biltmore is live." `app/api/wallboard/route.ts` only fetches the fallback's status when the primary isn't live (sequenced after the main `Promise.all`, since it depends on that result) — most of the time this second request never happens at all.
- `DatabaseFrame` and `LiveStreamPanel` (`app/wallboard/page.tsx`) are wrapped in a `.database-column` div (`grid-column: 3 / 5; grid-row: 1 / 4` in `app/globals.css`) instead of each setting its own `grid-row` directly on the shared `wall-grid` track. `.database-column` is its own nested grid (`grid-template-rows: 1.05fr 1fr`) so this pair's height ratio can be tuned independently of `hero-panel`/`geo-panel`, which still size off the shared 3-row track. This replaced an earlier layout where `.database-panel` spanned rows 1-2 (~77% of the column) and `.livestream-panel` got only row 3 (~23%) — the dashboard didn't need that much room (its scaled iframe just floated centered in the excess space via `.frame-shell`'s `justify-content: center`, showing as dead space above the chart) while the live video was squeezed thin (fixed 2026-07-05). If a taller `DATABASE_FRAME_VIEWPORT_HEIGHT` starts clipping the dashboard, raise `.database-column`'s first `fr` value rather than reverting to shared-row placement.
- Weather and traffic cameras live in the nested `.ops-column` (`grid-column: 5 / 6; grid-row: 1 / 4`); Traffic Detail moved out of that column on 2026-07-09 and now occupies the former right half of the Website Pulse top-left area (`grid-column: 2 / 3; grid-row: 1`). Website Pulse itself is the left half (`grid-column: 1 / 2; grid-row: 1`) and intentionally only contains the two GA realtime squares. The camera panel uses the freed `.ops-column` height for a `2 x 4` camera grid, so if future local-ops panels need more room, tune `.ops-column` row fractions/minimums first before shrinking the database/live stream column.

## Frontend Direction

- Use the DOT-inspired console system adopted from `cctv-weather-wall`: Overpass/Overpass Mono typography, near-black raised panels, amber for command/priority accents, cyan for instruments/HUD brackets, green only for live/nominal signals, and red only for failures. Keep the blueprint grid, restrained CRT scanlines/vignette, one-shot boot flash, and HUD corner brackets subtle enough for long unattended viewing.
- Maintain stable 16:9 TV layout first. Desktop and 4K viewports matter more than phone ergonomics.
- Keep text inside panels compact and readable. Avoid feature explanation copy inside the wallboard.
- Use icon buttons where possible and keep controls minimal for kiosk use.

## Testing Expectations

- Run `npm run typecheck` and `npm run build` before handoff.
- **Never run `npm run build` while `npm run dev` is also running against this repo** — both write to `.next` and will corrupt each other, breaking the dev server with `MODULE_NOT_FOUND` errors. Stop `dev` first (or skip `build` and rely on `typecheck` + the dev server's hot-reload) if a dev server is already up.
- **The user typically runs `npm run dev` as their actual local site, not as disposable test infrastructure.** Don't stop a dev server you didn't start yourself without asking, and leave `npm run dev` running when you finish a change rather than tearing it down.
- Visually inspect `/wallboard` at `1920x1080`; inspect `3840x2160` when practical.
- Also inspect the Cloudflare/static surface (`/index.html`, or `/` through `npm run dev:cloudflare`) because its camera player and CSS are intentionally maintained separately from the React page.
- Test both demo mode and configured env mode when credentials are available.
- For audio, verify the visual alert still works when browser autoplay blocks sound.
- When touching `lib/analytics.ts` or `lib/apify.ts` (or adding a new external call), spot-check that request volume is reasonable — for GA, check the Account data API quota history page in Google Analytics admin; for Apify, check the Apify Console's usage/billing page, since every call is real billed compute, not just a rate-limited free API.

## Known Environment Issue: `next dev` hangs on Node v24

This machine's only installed Node (`/usr/local/bin/node`, no nvm/brew present as of 2026-07-08) is **v24.14.0** — far newer than what Next.js 14.2.x (this project's version) is built/tested against. Symptom: `npm run dev` prints `✓ Starting...` (sometimes even binds the port successfully) but then hangs forever — every request times out with **zero CPU usage** on the `next-server` process (confirmed via `sample`/`ps`), i.e. it's genuinely blocked, not slow-compiling. This reproduces from a clean `.next` and with telemetry already disabled, so it isn't stale-build corruption or a first-run telemetry prompt. The `@next/swc-darwin-arm64` native binary itself loads and responds fine in isolation, so the hang is somewhere in Next's dev-server request pipeline under Node 24, not a broken native module per se.
- **Fix**: run this project on a supported LTS Node (18.18+ / 20.9+ / 22.x) instead of v24. Install `nvm` (or another version manager) and `nvm install 20 && nvm use 20` in this directory, ideally pinned via a `.nvmrc`. Do not attempt to silently install/switch global Node tooling without checking with the user first — it can affect their other projects.
- If `npm run dev` seems stuck again in the future, check `node -v` before re-diagnosing from scratch.
