# Agent Notes

## Project Intent

This repo owns a private cinematic monitoring room wallboard for a single landscape TV. The first screen should be the working wallboard experience, not a marketing page. Keep the interface operational, dense, dark, readable from across the room, and free of visible organization branding.

## Implementation Conventions

- Prefer configuration-driven panels and server-side data fetching.
- Keep `/wallboard` suitable for unattended Apple TV signage/browser use.
- Preserve graceful degradation: missing credentials, blocked embeds, or API failures should show setup/degraded states rather than crash.
- GA traffic sources should use a standard GA report such as today's `sessionDefaultChannelGroup`; GA realtime rejects source/medium dimensions.
- The world access map uses GA realtime `countryId` plus `country`; add coordinates to `COUNTRY_COORDINATES` in `lib/analytics.ts` for new countries that need precise placement.
- Iframe dashboards are fragile. Check vendor support for kiosk/embed URLs before assuming a normal logged-in page can render inside the wallboard.
- The Site24x7 embed is scaled through `DATABASE_FRAME_VIEWPORT_WIDTH` and `DATABASE_FRAME_VIEWPORT_HEIGHT`; use the current `1024x640` default unless visual QA shows clipping.
- The `DatabaseFrame` iframe's auto-scale calculation (`updateScale` in `app/wallboard/page.tsx`) only has to reserve the `.frame-shell`'s own padding now — the "All Monitors" / "Website Check" / "SSL" stat boxes (`.database-frame-summary`) and the ticker were both removed from this column (2026-07-05; the ticker moved to a full-width `.ticker-bar`, see below). If a new element is ever added inside `.frame-shell` below the iframe, its rendered height must go back into this reservation math (via a ref + `ResizeObserver`, as the old summary/log entries did) or the iframe will overlap/clip it.
- Site24x7 public dashboard dark theme is configured in Site24x7 share settings, not by styling the iframe from this app.
- Do not embed YallBot/YouTube for the ambient "emergency operations center" feel — that's realized natively instead, via the `SystemLog` component (real alerts + latest Instagram/Facebook posts + heartbeat lines) in `app/wallboard/page.tsx`; treat that as the actual implementation of the style inspiration, not a placeholder for a future embed. `SystemLog` renders as a horizontally scrolling ticker (`.ticker-track`/`.ticker-entry` in `app/globals.css`), not a stacked list — the underlying `useSystemLog` hook logic (entry state, dedup, ambient rotation) is unchanged, only the render/CSS layer is a marquee. Scroll speed is held roughly constant regardless of content length by measuring `scrollWidth` and deriving `animationDuration` from a fixed px/sec constant (`TICKER_PX_PER_SECOND`), not a fixed duration. The ticker itself lives outside `wall-grid` now, in a full-width `.ticker-bar` between the grid and `wall-footer` (moved 2026-07-05 from inside the Active Database System panel, where it and the removed stat boxes were cramping the dashboard iframe) — `useSystemLog(payload)` is still computed once at the `WallboardPage` level and passed straight to `<SystemLog>` there; `DatabaseFrame` no longer touches it. `.wall-grid`'s `height: calc(100vh - Npx)` (both the base rule and the shrunk-viewport breakpoint) reserves `.ticker-bar`'s margin + height on top of the header/incident/footer reservation it already accounted for — keep those two numbers in sync if `.ticker-bar`'s sizing changes. This is distinct from the `LiveStreamPanel` (below), which does embed YouTube directly — that panel shows the org's actual live broadcast for a real operational reason (is the service airing right now), not ambient decoration, so the "no embed" rule doesn't apply to it.
- The ticker's ambient content used to be Hacker News + Fox News headlines (`lib/newsFeed.ts`, both free/no-key public feeds); that was fully replaced 2026-07-05 with the church's own latest Instagram/Facebook posts per an explicit request, and `lib/newsFeed.ts` was deleted (along with the now-unused `fast-xml-parser` dependency) rather than left as dead code. Neither Instagram nor Facebook has a free public feed — confirmed by direct testing that anonymous requests get nothing usable (Facebook returns a flat error page, Instagram returns an empty JS shell) — so this goes through Apify (`lib/apify.ts`, `APIFY_TOKEN`, the `apify~instagram-post-scraper` and `apify~facebook-posts-scraper` actors' `run-sync-get-dataset-items` endpoint), which runs the actual scraping on its own infrastructure under its own ToS relationship with those platforms; this app never scrapes them directly or automates a personal login. Each call is real billed compute, so the cache is 15 minutes per platform (`CACHE_MS` in `lib/apify.ts`) — don't shorten without accounting for cost. A newly-detected post (tracked client-side via `seenSocialPostIds`, same pattern as `seenAlertIds` for real alerts) surfaces immediately as a "New post" ticker line at `watch` severity rather than waiting for the 2-5 minute ambient rotation — this is still ticker-only and never touches `payload.alerts` or the audible-alert system.
- The "serious" alarm (`database-monitors-down` only — i.e. the Site24x7 All Monitors feed reporting a down monitor) plays a distinct square-wave siren tone (`playAlarmPulse`, `app/wallboard/page.tsx`) on a fixed 12s repeat with no cooldown gate, separate from the normal `playTone`/180s-cooldown path used by every other critical alert (website health, SSL, traffic zero/spike/drop). It stops automatically once `database-monitors-down` drops out of `payload.alerts` — there is no acknowledge/silence control, and it still requires the same `arm audio` gesture as the rest of the audio system. The repeat-siren `useEffect` must key off a stable boolean (`seriousAlertActive`), not the `alerts.find(...)` object itself — every 30s poll (`POLL_MS`) produces a brand-new `alerts` array/object even when nothing changed, so depending on the object reference tears down and restarts the `setInterval` every poll, breaking the fixed 12s cadence into an irregular 12/12/6s pattern (fixed 2026-07-05).
- Map dots must be keyed by more than coordinates: `mapPointKey` (`app/wallboard/page.tsx`) includes `city ?? region` alongside `countryCode`/lon/lat because several distinct GA realtime regions can legitimately share one fallback coordinate — any city not present in `COUNTRY_COORDINATES` collapses to the same country-level anchor (see the world-map bullet above). Keying on coordinates alone let a `Map` silently collapse multiple distinct regions onto a single flickering dot, losing all but the last one in poll order (fixed 2026-07-05, confirmed live with four Biltmore-area cities sharing the US country anchor).
- **Any new external API call (GA, Apify, or future additions) must follow the caching discipline in `lib/analytics.ts` / `lib/apify.ts`: a generous TTL cache, in-flight-promise de-duplication so concurrent requests share one outstanding fetch, and a silent fallback to the last-good/empty result on failure — never throw, never surface a raw provider error to the UI.** This was learned the hard way: an early version polled GA Realtime aggressively enough to burn ~14,000 tokens in a single clock-hour and exhaust the quota. GA's Realtime token quota resets on the clock hour, not a rolling window from the failure — any quota-style backoff should target the next hour boundary, not a fixed duration.
- Panels with a fixed-height budget that can shrink (e.g. via a parent grid's `fr` sizing) should gracefully hide non-essential sub-content rather than render an unreadably squeezed sliver — see `.hero-panel`'s `@container` query in `app/globals.css`, which hides the trend sparkline below 275px of panel height instead of showing ~2px bars. Prefer this pattern (CSS container queries scoped to the panel, not viewport media queries) for any future panel content that depends on space freed up by other layout changes.
- The same rule applies to text sizing, not just visibility: never size text with `vw` inside deeply nested grid layouts. The `.mini-metrics strong` numbers under Website Pulse used `clamp(21px, 2.2vw, 40px)`, which scales off total viewport width even though that box sits four grid levels deep (`wall-grid` > `hero-panel` > `metric-row` > `mini-metrics`) — on wide/4K screens the numbers kept growing while the actual box didn't, spilling past the box border. Fixed by giving `.mini-metrics div` `container-type: inline-size` and sizing `strong` off `cqw` (the box's own rendered width) instead, plus `overflow: hidden` on the box as a safety net (fixed 2026-07-05).
- `LiveStreamPanel` (`app/wallboard/page.tsx`) shows the org's YouTube channel (`YOUTUBE_LIVE_CHANNEL_HANDLE`) live in a muted, autoplaying embed when it's actively broadcasting, with a "LIVE" badge; otherwise a quiet "Not currently live" state. Live/offline detection (`lib/youtubeLive.ts`) needs no YouTube Data API key: it scrapes the channel's `/<handle>/live` page and checks whether the `<link rel="canonical">` tag points at a `watch?v=` URL (only true while actually live), cached 45s with the same in-flight-dedup/silent-fallback discipline as every other external call in this repo (see the bullet below). Use the `@handle` path, not `/channel/<id>/live` — confirmed via direct curl that the channel-ID path is unreliable for at least one real channel (stayed on the channel page while genuinely live) while the handle path resolved correctly for every channel tested. Missing `YOUTUBE_LIVE_CHANNEL_HANDLE` means the panel doesn't render at all (`payload.liveStream.enabled === false`), not a setup warning — same convention as `DATABASE_MONITORS_STATUS_URL`. The embed is muted via the `mute=1` URL param baked into the iframe `src`, not the Web Audio `arm audio` gate used elsewhere — it's ambient/visual-only and was never routed through that system. It originally occupied the one cell the shared 3-row `wall-grid` left open below the database panel, but that gave it only ~23% of the column's height against the database panel's ~77% — see the `.database-column` note below for why that changed (2026-07-05).
- If the primary channel isn't live, `LiveStreamPanel` checks a second channel (`YOUTUBE_FALLBACK_CHANNEL_HANDLE`, defaults to `@livenowfox`) and shows it instead if that one's live, with a distinct amber `.livestream-badge.fallback` badge reading "live now — [source]" so it never reads as "Biltmore is live." `app/api/wallboard/route.ts` only fetches the fallback's status when the primary isn't live (sequenced after the main `Promise.all`, since it depends on that result) — most of the time this second request never happens at all.
- `DatabaseFrame` and `LiveStreamPanel` (`app/wallboard/page.tsx`) are wrapped in a `.database-column` div (`grid-column: 3 / 5; grid-row: 1 / 4` in `app/globals.css`) instead of each setting its own `grid-row` directly on the shared `wall-grid` track. `.database-column` is its own nested grid (`grid-template-rows: 1.05fr 1fr`) so this pair's height ratio can be tuned independently of `hero-panel`/`geo-panel`, which still size off the shared 3-row track. This replaced an earlier layout where `.database-panel` spanned rows 1-2 (~77% of the column) and `.livestream-panel` got only row 3 (~23%) — the dashboard didn't need that much room (its scaled iframe just floated centered in the excess space via `.frame-shell`'s `justify-content: center`, showing as dead space above the chart) while the live video was squeezed thin (fixed 2026-07-05). If a taller `DATABASE_FRAME_VIEWPORT_HEIGHT` starts clipping the dashboard, raise `.database-column`'s first `fr` value rather than reverting to shared-row placement.

## Frontend Direction

- Use high-contrast cinematic data-center styling: dark base, cyan/green signal colors, sharp panel lines, subtle scanline/grid motion.
- Maintain stable 16:9 TV layout first. Desktop and 4K viewports matter more than phone ergonomics.
- Keep text inside panels compact and readable. Avoid feature explanation copy inside the wallboard.
- Use icon buttons where possible and keep controls minimal for kiosk use.

## Testing Expectations

- Run `npm run typecheck` and `npm run build` before handoff.
- **Never run `npm run build` while `npm run dev` is also running against this repo** — both write to `.next` and will corrupt each other, breaking the dev server with `MODULE_NOT_FOUND` errors. Stop `dev` first (or skip `build` and rely on `typecheck` + the dev server's hot-reload) if a dev server is already up.
- **The user typically runs `npm run dev` as their actual local site, not as disposable test infrastructure.** Don't stop a dev server you didn't start yourself without asking, and leave `npm run dev` running when you finish a change rather than tearing it down.
- Visually inspect `/wallboard` at `1920x1080`; inspect `3840x2160` when practical.
- Test both demo mode and configured env mode when credentials are available.
- For audio, verify the visual alert still works when browser autoplay blocks sound.
- When touching `lib/analytics.ts` or `lib/apify.ts` (or adding a new external call), spot-check that request volume is reasonable — for GA, check the Account data API quota history page in Google Analytics admin; for Apify, check the Apify Console's usage/billing page, since every call is real billed compute, not just a rate-limited free API.
